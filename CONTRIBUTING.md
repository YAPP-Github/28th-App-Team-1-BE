# CONTRIBUTING

## 브랜치 전략 (GitHub Flow)

`main` 브랜치 하나만 영구 유지합니다. 모든 작업은 이슈를 생성한 뒤 브랜치를 따서 진행하고, PR 머지 후 브랜치를 삭제합니다.

```
main
  └── feat/{이슈번호}-{기능명}
  └── fix/{이슈번호}-{버그명}
  └── chore/{이슈번호}-{내용}
```

버전은 `main` 브랜치에 태그로 관리합니다. (`v1.0.0`, `v1.1.0`, ...)

## 이슈 템플릿

| 템플릿 | 브랜치 prefix | 용도 |
|---|---|---|
| Feature | `feat/` | 새 기능 구현 |
| Fix | `fix/` | 버그 수정 |
| Chore | `chore/` | 빌드·설정·리팩토링·문서·테스트 등 |

## 커밋 메세지 컨벤션

```
<type>: <변경 요약 (한글 또는 영문, 50자 이내)>
```

| 타입 | 용도 |
|---|---|
| `feat` | 새로운 기능 |
| `fix` | 버그 수정 |
| `refactor` | 기능 변경 없는 코드 개선 |
| `docs` | 문서 작성·수정 |
| `test` | 테스트 추가·수정 |
| `chore` | 빌드·설정 등 기타 |

- 브랜치 prefix와 커밋 타입은 독립적입니다. `feat/` 브랜치 안에서 `fix:` 커밋을 올릴 수 있습니다.
- 커밋 메세지에 이슈번호를 포함하지 않습니다.

## PR

PR을 올릴 때 관련 이슈를 `closes #번호` 로 연결하면 머지 시 이슈가 자동으로 닫힙니다.
